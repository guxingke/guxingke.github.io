Spring 框架是工厂模式的集大成者。从工厂的角度，简述 Spring 框架的核心。

  • 生产什么？Bean, 没有贴切的中文翻译。
  • 原材料是什么？ BeanDefinition
  • 原材料放在哪儿？ BeanDefinitionRegistry
  • 原材料如何获得？工厂成立时通过配置告知，主流两种方式，基于 xml 的配置和基于 annotation 的配置。
  • 工厂的工作流程？
    1. 工厂初始化
    2. 收集原材料 BeanDefinition
    3. 按需生产 Bean, 工厂开工后优先生产需要立即生产的 Bean (lazy=false)

处理 BeanDefinitionPostProcessor

此类型的处理器可以生成新的 BeanDefinition, 一些外部框架与 Spring 集成时会用到该处理器，在 Spring 内部，ConfigurationClassPostProcessor 是个典型实现。

处理逻辑如下：

  1. 先处理方法入参的 processors， 这一步可能会生成新的 BeanDefinitionPostProcessor 实例。
  2. 按优先级，依次处理实现了 PriorityOrdered，Ordered 接口的 BeanDefinitionPostProcessor 实例。
  3. 处理其他的 BeanDefinitionPostProcessor 实例, 由于每次处理都有可能产生新的 BeanDefinitionPostProcessor 实例, 这一步是通过循环来处理的，确保所有的 BeanDefinitionPostProcessor 实例都被处理过。

处理其他的 BeanFactoryPostProcessor

到这一步时，所有的 BeanDefinition 已经生成完毕，并且没有 bean 被实例化，在这一步可以修改 BeanDefinition ，甚至可以实例化 bean。

ConfigurationClassPostProcessor 与 ConfigurationClassParser

Xml 配置基本告别历史舞台，新的应用都是基于 Annotation 配置的，在 Annotation 配置时，@Configuration 注解是绕不过去的，而这两个类是处理 @Configuration 注解的核心。

ConfigurationClassPostProcessor 类

  1. 实现了 BeanDefinitionPostProcessor
  2. 实现了 PriorityOrdered

SpringBoot 自动配置原理

从 @SpringBootApplication 注解为切入点，该注解是个组合体，集合了一堆注解，包括：

  • @SpringBootConfiguration 该注解组合了 @Configuration 注解
  • @EnableAutoConfiguration 自动配置的核心注解
    • @AutoConfigurationPackage
    • @Import(AutoConfigurationImportSelector.class) 导入了自动配置 Selector
  • @ComponentScan

核心方法 AutoConfigurationImportSelector.AutoConfigurationGroup#process

  1. 通过 SpringFactoriesLoader 加载 key 为 org.springframework.boot.autoconfigure.EnableAutoConfiguration 的自动配置实例
  2. 过滤掉重复的和启动类上通过 exclude 注明的类
  3. 自动配置类初始化并且按需注入依赖的变量（context,environment 等）

通过 process 把自动配置的 Configuration 识别出来。

执行自动配置

自动配置的目标是把符合条件的 BeanDefinition 识别出来，放到 BeanDefinitionRegistry 里。 通过自动配置类的 @Conditional 注解内容生成 Condition, 判断 Condition 是否匹配，匹配则添加，不匹配则忽略。

其他

  1. 自动配置触发时机，作为 BeanDefinitionRegistryPostProcessor 接口的实现 ConfigurationClassPostProcessor 类在 AbstractApplication#invokeBeanFactoryPostProcessors 阶段被启动。
